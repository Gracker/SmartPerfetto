⚠️ 数据质量提示：

  • R008_TTID_GT_DUR：TTID (1912ms) > dur (1339ms)，差值 573ms 被启动窗口后的 MQ_Chain_*_Step 模拟任务 slice 占据（18 个，各 ~7-8ms），无 RenderThread/SurfaceFlinger 活动。TTID 不可信，分析窗口基于 dur。
  • R009_TYPE_RECLASSIFIED：Perfetto 原始分类为冷启动，存在 bindApplication slice 确认为冷启动，无矛盾。
  • blocked_functions 不可用：主线程状态分布的 blocked_functions 全为 "-"（内核未开启 CONFIG_SCHEDSTATS），阻塞根因基于 hot_slice_states 的 per-slice 状态分布间接推断。

2. 关键发现

[HIGH] Activity 初始化阶段模拟负载耗时 710ms (self=250ms, 18.7%) ← 根因 A4/C2: 测试负载 CPU-bound + 阻塞混合

  • 描述：LoadSimulator_ActivityInit slice wall time 710ms（占启动 53%），self_ms=249.8ms（18.7%）
  • 根因推理链：
    ① 四象限显示 Q1=62.8%（主线程 841ms 大核 Running），Q4b=35.1%（470ms Sleeping）→ CPU-bound + 阻塞混合模式
    ② hot_slice_states 显示 LoadSimulator_ActivityInit 内部：Running=447ms(63%) + S=250ms(35.2%) + D=11ms(1.6%)
    ③ 其子 slice SimulateInflation (178ms) 98.4% Running → 纯 CPU-bound 模拟
    ④ RealInflation (104ms) 99.5% Running → 真实布局 inflate 也是 CPU-bound
    ⑤ S 状态 250ms 来自模拟任务内部的锁/等待，无 blocked_functions 可定位具体阻塞源
  • SR 交叉验证：SR01-SR20 无命中（测试负载不属于标准慢启动模式）
  • 结论：此 slice 慢的根因是测试框架注入的模拟计算负载（CPU-bound）+ 模拟阻塞等待（线程等待锁/IO/Binder），不是真实应用问题
  • 建议：如需评估真实应用启动性能，应使用生产环境应用替换测试负载

[HIGH] ChaosTask 模拟负载耗时 479ms (self=456ms, 34.1%) ← 根因 C2: 测试负载 CPU-bound

  • 描述：ChaosTask 共 70 次调用，累计 wall time 479ms，self_ms=456ms（34.1%）
  • 根因推理链：
    ① hot_slice_states 显示 ChaosTask 每次调用 100% Running 状态
    ② 平均每次 ~6.5ms，最大 16ms，纯 CPU 计算
    ③ 无 S/D 状态，无阻塞
  • 结论：纯 CPU-bound 模拟负载，70 次调用累积 456ms
  • 建议：测试框架的 ChaosTask 是人为注入的 CPU 压力

[HIGH] App 初始化阶段模拟负载耗时 478ms (self=207ms, 15.5%) ← 根因 C2: 测试负载

  • 描述：LoadSimulator_AppInit wall time 478ms（占启动 35.7%），self_ms=207ms（15.5%）
  • 根因推理链：
    ① 位于 bindApplication 阶段，是 LoadSimulator_ActivityInit 的对应 App 层模拟
    ② hot_slice_states 未单独列出（被包裹在 bindApplication 内），但从全局 S=470ms(35%) 推断存在阻塞成分
  • 结论：测试框架在 Application 初始化阶段注入的模拟负载

[MEDIUM] 主线程 Q4(Sleeping) = 35.1% (470ms) — 阻塞来源待确认

  • 描述：主线程有 470ms 处于 Sleeping 状态，占启动时长 35.1%
  • 根因推理链：
    ① 全局 blocked_functions 为 "-"，无法直接定位阻塞函数
    ② hot_slice_states 显示 LoadSimulator_ActivityInit 内 S=250ms(35.2%)，clientTransactionExecuted 内 S=256ms(30.3%)
    ③ 同步 Binder 总量 <5ms，可排除 Binder 阻塞
    ④ GC 仅在后台线程（HeapTaskDaemon），主线程 GC 锁竞争 <0.01ms，可排除 GC 阻塞
    ⑤ 推断：S 状态主要来自测试框架注入的模拟等待/锁（如 ChaosTask 调用间隔、模拟 IO 等待）
  • 结论：基于间接证据推断，blocked_functions 不可用。S 状态主要由测试负载模拟产生，非真实阻塞
  • 建议：后续抓 trace 时启用 echo 1 > /proc/sys/kernel/sched_schedstats 以获取完整 blocked_functions

[MEDIUM] R008: TTID 差值 573ms 去向分析

  • 描述：TTID=1912ms > dur=1339ms，差值 573ms
  • 分析：查询 end_ts 到 TTID 时间段，发现 18 个 MQ_Chain_*_Step slice（各 ~7-8ms），全部在主线程执行，总计 ~132ms。剩余 ~441ms 无显著 slice 活动（可能是空闲/等待 VSync）
  • 结论：TTID 不可信。差值由启动窗口后的模拟任务延续 + 框架 TTID 计算时机导致

[LOW] Binder 线程池仅 1 个线程 — 当前非瓶颈但存在潜在风险

  • 描述：Binder 线程池利用率 100%（Running 6ms / Idle 0ms），但主线程同步 Binder 总量 <5ms
  • 结论：当前 Binder 用量极低，1 个线程足够。但线程池过小在高并发场景下可能成为瓶颈
